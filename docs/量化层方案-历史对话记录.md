## 对话 1：量化因子框架初步方案

### 用户提问

既然已经拿到日k的数据了，那么量价数据已经有了，如果要开始做量化因子挖掘，策略，回测，当前的项目结构，怎么开发，比如目录结构，应用的技术等等，考虑拓展性，比如现在只有量价因子，未来会有其他类型的比如财报基本面等。

### Claude 回复

推荐**自研轻量框架，接现有 ETL/Service 分层**，而不是引入 qlib / backtrader。理由：现有分层（CLI → Strategy → Workflow → Extract/Transform/Load）已经把"批量算 + 入库"这套模式磨得很顺，因子计算本质就是"读量价 → 算 → 入因子值表"，套同一套模板最省事；qlib 框架重、parquet 范式与你的 PG 范式割裂，backtrader 是事件驱动循环，跟"截面调仓多因子"路线不契合。

**建议目录骨架（新增 `src/research/` 域）：**

```
src/
  research/
    factor/              # 因子定义：BaseFactor + 注册表
      price_volume/      # 量价因子（动量/波动率/反转/成交量…）
      fundamental/       # 财报因子（未来：ROE/营收增速…）
      sentiment/         # 另类因子（未来）
    dataset/             # 统一数据访问层（屏蔽 SQL，回测/因子都走它）
      kline.py / factor.py / universe.py
    backtest/            # 截面调仓回测引擎 + 组合管理 + 手续费滑点
    strategy/            # 因子→信号→仓位（单因子排序、多因子合成）
    performance/         # IC/RankIC/IR、夏普、最大回撤
  entities/data_entities/
    factor_value_entities.py    # (factor_name, ts_code, trade_date, value) 长表
    factor_meta_entities.py     # 因子元数据（频率/依赖/版本）
  etl/strategy/factor/   # 复用现有 ETL 模式：批量算因子 → upsert factor_value
  api/routers/research.py # 未来 Admin 看因子值/回测结果
  web/admin/.../pages/research/  # 因子库页 + 回测结果页
```

**关键技术取舍：**
- 计算：先 pandas + numpy；性能瓶颈再上 polars / numba（**别一开始就过度优化**）
- 存储：因子值入 PG 长表 `factor_value`，复用已验证的 `bulk_upsert_postgresql` + 95% 完整性快照那套（→ 因子也能跑 `check-complete`）
- 截面回归：statsmodels；多因子合成：sklearn linear/ridge 或自己实现 IC 加权
- BaseFactor 接口约定 `frequency`（daily/weekly/quarterly）+ `dependencies`（kline / report / ...），未来财报因子按 `report_period → trade_date` forward-fill 即可复用同一张因子表

**最小闭环里程碑（建议按这顺序）：**
1. `factor_value` 表 + 1 个 demo 因子（20 日动量） + ETL `factor compute` 命令
2. 极简截面回测：每月底按因子排序分 10 组、等权持仓、算下期收益 → 输出 IC + 分组净值
3. 上 Admin 一个因子结果页

## 对话 2：引入分钟线 + 事件驱动后的重新设计

### 用户补充

关于k线，现在只有日线未来会有分钟线，我本身也是要 Parquet + DuckDB 存储的，同时我也会有事件驱动，比如财报发布、资讯新闻等产生。

### Claude 回复（完整重新设计）

这三条把方案推到完全不同的形态。重新整理推荐：

#### 存储双层（最关键的决策）

| 层 | 用途 | 内容 |
|---|---|---|
| **PG** = OLTP/真理源 | 强一致、易 upsert、Admin 直读、事务 | 日线、财报、停复牌、交易日历、stock_list、各种 period_count 快照、missing_log、新闻事件元数据、**因子元数据** |
| **Parquet + DuckDB** = OLAP/研究仓库 | 列存、谓词下推、批量扫描快、零服务 | **分钟线、因子值、回测中间集、tick** |

**分钟线一开始就别进 PG**，5500 股 × 240 min × 250 日 ≈ 3.3 亿行/年，PG 单表能扛但备份/扫描/迁移都痛苦；直接落 `parquet/kline_minute/dt=YYYYMM/symbol=XX/part-*.parquet` 分区。

**DuckDB 角色**：本地嵌入式 OLAP 引擎，`attach` PG（postgres_scanner 扩展）+ `read_parquet` glob，一条 SQL 跨源 join（如分钟线 ⨝ 财报披露日 ⨝ 因子值），研究阶段不用写胶水代码。

**同步管道**：ETL Strategy 写完 PG → finalize 阶段同步 dump 到 Parquet（也可以反过来：分钟线直接落 Parquet，PG 只放 `kline_minute_period_count` 快照保持你"95% 完整性"那套 ops 模式）。

#### 计算双引擎（别想统一）

| 引擎 | 场景 | 实现 |
|---|---|---|
| **截面调仓**（向量化） | 周/月调仓多因子、IC 测试 | DuckDB SQL + Polars，全表算到底 |
| **事件驱动** | 财报披露后漂移、新闻冲击、分钟级 intraday | 自研事件循环：`heapq[(timestamp, event_type, payload)]`，多源 merge 消费 |

共享 `Portfolio` / `Cost` / `Universe` / `Performance` 模块即可，主循环不强求复用。

#### 事件统一抽象

`Event(ts, kind, ts_code?, payload)`，`kind ∈ {bar, report_announce, news, schedule, suspend}`：

- `bar`：分钟线、日线
- `report_announce`：财报披露事件（用 `report_*.ann_date` 而不是 `end_date`！这是事件驱动的关键，**披露才是市场反应起点**）
- `news`：资讯（量小，PG `event_news` 表 + JSON payload 够用）
- `schedule`：定时调仓

事件源各自有 reader（PG / Parquet），引擎按时间堆合并消费，策略只看 `on_event(event)`。

#### 调整后的目录骨架

```
src/
  research/
    storage/             # Parquet 分区写入 / DuckDB conn / PG↔Parquet sync
    dataset/             # 统一数据访问（屏蔽 PG vs Parquet）
      kline_daily.py     kline_minute.py     factor.py
      event.py           universe.py
    factor/
      price_volume/      fundamental/        event/    # 事件因子
      base.py            registry.py
    backtest/
      crosssection/      # 截面调仓（DuckDB+Polars）
      event_driven/      # 事件驱动（heapq 主循环）
      common/            # Portfolio / Cost / Slippage
    strategy/
    performance/         # IC / RankIC / IR / Sharpe / MDD
  entities/data_entities/
    factor_meta_entities.py    # 因子元数据（PG）
    event_news_entities.py     # 新闻原始落 PG
  etl/strategy/
    kline_minute/        # 分钟线 → Parquet
    factor/              # 因子计算 → Parquet
    news/                # 资讯抓取 → PG event_news
  api/routers/research.py
```

#### 技术栈取舍

- **Polars over pandas**（分钟级因子）：lazy + Rust 后端，比 pandas 快 5-20x，API 也更干净
- **pyarrow** 写 Parquet（dataset API 支持分区、谓词下推）
- **DuckDB** 0.10+：原生 PG scanner，Parquet glob，HNSW 索引（未来做新闻向量检索）
- **不要 qlib**：它的 `.bin` 二进制格式锁死你的存储路线；它的因子 DSL 也不利于事件因子表达
- **事件驱动**：先自研（200 行内能跑），分钟级真撞性能瓶颈再考虑 Rust binding / vnpy

#### 最小推进顺序（按风险/价值排）

1. **`research/storage/` 起步**：定 Parquet 分区规范 + DuckDB conn 工厂 + PG→Parquet 日线 dump（验证存储双层链路，最小代价拿到 DuckDB 跨源查询能力）
2. **分钟线 ETL → Parquet**（你的下一个数据源，直接走新路）
3. **`research/factor/` + 1 个 demo 日频因子**（动量），值落 Parquet，验证 Polars + DuckDB 算法链路
4. **`research/backtest/crosssection/`**：月度调仓 IC + 分组净值，闭环
5. **`research/backtest/event_driven/`**：以"财报披露后 N 日漂移"为第一个事件策略（数据已有 → 验证事件抽象 → 顺便测新闻接入）
6. 新闻 ETL + 事件因子

## 对话 3：ETL 分层设计 - PG 导出 Parquet

### 用户提问

如果在 etl 里增加一个功能把 pg 的日 k 数据存到 duckdb 中，是写在 load 层么，然后在 strategy 策略层直接调用 load？还是写在 transform 里先做转换？那数据从 pg 抽取出来是在 extract 的 load 里？

### Claude 回复

| 问题 | 答案 | 理由 |
|---|---|---|
| 写在 Load 层，Strategy 直接调？ | **不**。走完整 Extract→Transform→Load | PG 全表读必须分批 yield（按 month or 100w 行）防 OOM；要补分区列；将来加字段过滤没地方放 |
| 要 Transform？ | **要，但很薄** | 列裁剪、PG `numeric` → arrow `double`、补 `year_month` 分区键 |
| PG 抽数放 Extract 还是 Load？ | **Extract** | ETL 分层的语义是「读/转/写」，不是「源在哪」。PG 在这条链路里就是数据源 |

**关键判断：这是新子域，不要污染现有 kline 链路**

现有 `src/etl/{extract,transform,load}/kline/` 都是为 Tushare/tdx → PG 设计的，硬塞 PG → Parquet 会让职责双向纠缠。新建 `warehouse` 子域：

```
src/etl/
  extract/warehouse/
    kline_daily_pg_extract.py      # batch_yield_by_month / get_max_dumped_date
  transform/warehouse/
    kline_daily_warehouse_transform.py   # 列裁剪 + 类型 cast + 加分区列
  load/warehouse/
    parquet_load.py                # 通用：分区写、overwrite-by-partition
    kline_daily_parquet_load.py    # K 线特化：分区路径约定
  workflow/warehouse/
    kline_daily_warehouse_workflow.py
  strategy/warehouse/
    kline_daily_warehouse_strategy.py
```

**关键设计点：**

1. **分区策略**（一次定终生）：`parquet_root/kline_daily/dt=YYYYMM/part-{uuid}.parquet`，按月分区
2. **增量起点**：Parquet 自身 dt 分区就是状态，不维护单独 state 表
3. **完整性守门**：复用现有 ops 模式
4. **Overwrite-by-partition 幂等**：先删 `dt=YYYYMM/` 目录，再写
5. **DuckDB 不参与 dump 写入**：只在研究/回测/校验阶段用

## 对话 4：分区粒度分析

### 用户提问

首先你分析下 parquet 是一个月一个文件，还是一个股票一个文件那个在后续效率更高，日线场景下。

### Claude 分析

| 维度 | 按月 `dt=YYYYMM/` | 按股 `symbol=XX/` |
|---|---|---|
| 文件数（30 年） | 360 | 5500 |
| 单文件大小 | ~10MB（甜区） | ~2MB（偏小，IO 放大） |
| **截面查询** | 1 个文件 row-group prune | **扫 5500 个文件读 1 行**（灾难） |
| **区间扫描** | 连续 N 个月，顺序 IO | 5500 文件并行 filter，元数据开销大 |
| 单股全历史 | 360 文件 × row-group prune | 1 文件直读 |
| **每日增量** | 重写当月分区（10MB） | **更新 5500 个文件**（写放大爆炸） |

**结论：按月分区**。日线场景主路径是截面 + 区间扫描，按月分区在三种查询中只有「单股全历史」略弱一点，代价可接受；按股分区在截面和增量写两个最重要场景都是灾难。

**优化**：写入时按 `(trade_date, ts_code)` 排序，让每个 row group 的 trade_date min/max 紧凑 → DuckDB 截面查询能精确跳过非匹配 row group。
